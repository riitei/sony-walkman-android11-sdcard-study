# Android 11+ DAP SD 卡存储行为研究：以 Sony Walkman 为例

[返回主 README](../README.md) · [繁體中文](README.zh-TW.md) · [English](README.en.md) · [FAQ](faq.md) · [相容性矩阵](compatibility.md)

## 前言

这篇文章不是单纯记录“QQ 音乐怎么搬到 SD 卡”，而是要回答一个更底层的问题：

> 在 Android 11 之后，外接 SD 卡究竟能不能被 DAP（数字音频播放器）上的 App 真正当作主要可用存储空间？

Sony Walkman、iBasso、HiBy、FiiO 等 Android DAP，本质上都可能碰到同一类问题：

- 系统看得到 SD 卡，不等于 App 会把数据写进去
- App 本体搬得动，不等于离线歌曲也会跟着搬
- UI 显示容量变大，不等于主共享存储空间真的被整合

因此，本文虽然以 **Sony Walkman + Android 11 + QQ 音乐** 为实测案例，但真正要处理的是 **Android 11+ DAP 的存储分层问题**。

---

## 问题定义

这篇文章主要要解决的是以下几类实际问题：

1. **SD 卡明明已被系统识别，为什么 App 还是在占用内置存储？**
2. **App 明明可以搬移，为什么离线歌曲或缓存没有跟着搬？**
3. **为什么 Windows / MTP 还是只看到约 103GB，而不是完整 1TB？**
4. **哪些步骤只是研究探索，哪些才是普通用户可重现的方法？**
5. **在 non-root 前提下，到底有没有一条稳定、可验证、可长期使用的路线？**

本文的目标不是把所有现象混成一句话，而是把“系统层”“UI 层”“App 数据层”拆开。

## 核心结论先行

先讲结论，避免读者在前面步骤中混淆：

1. **Sony Walkman 的外接 1TB SD 卡可以被系统识别，也可以成为 App 可选择的存储位置。**
2. **QQ 音乐 App 本体可以通过系统 UI 手动搬移。**
3. **但 QQ 音乐的离线歌曲数据不一定跟着搬移。**
4. **本文最后确认的可用路线是：USB 调试 + ADB 验证 + Sony UI 手动搬移 App。**
5. **本文不把 root 当成前提，也不把重新格式化 SD 卡当成普通读者必做主流程。**

一句话总结：

> **App 本体搬移成功，不代表 App 下载数据搬移成功。**

## 研究目的

本次研究的目的有三个：

### 1. 解决容量限制
Sony Walkman 内置存储约 128GB，对高音质音乐、串流离线数据与长期使用来说偏小。希望让 1TB SD 卡成为主要音乐承载空间。

### 2. 解决写入压力集中在内置存储的问题
如果大量下载、删除、重建缓存都长期写在内置存储，内置空间会同时承受容量与写入压力。因此希望把主要音乐数据流量导向可替换的 1TB SD 卡。

### 3. 验证 Android 11 存储分层在 DAP 上的实际影响
直觉上，1TB SD 卡插进去之后，应该可以让音乐 App 直接使用大容量存储。但实际上，Android 11 的存储分层与 OEM 设计，可能让“可见容量”与“可实际使用容量”不同。

2. 开启 **USB 调试**
3. 使用 USB 线连接电脑

```bash

```

若显示 `unauthorized`，代表设备尚未完成授权，需回到设备端确认 USB 调试提示窗口并按下允许。

## 底层模型：先分清楚三层

在正式操作前，必须先把 Android 11 的存储结构拆开看。

### 1. 共享存储层
典型路径：

```text
/storage/emulated/0
```

这一层通常是：

- 用户在文件管理器中熟悉的“内部共享存储空间”
- App 最常显示给用户看的下载路径
- Windows 通过 MTP 最容易看到的那一层

很多人会误以为这一层就是全部存储，但它其实只是 App 和用户最常接触到的一层。

### 2. 扩展卷 / 外接卷层
典型路径：

```text
/mnt/expand/<UUID>
```

这一层通常是：

- 外接存储被系统挂载后的扩展卷路径
- Package Manager 可能把 App 本体导向这里
- 一般用户不一定能直接通过文件管理器操作

这一层是系统底层真正看到的外接卷。

### 3. App Storage 管理层
这一层不是单一路径，而是系统 UI 提供的管理入口，例如：

```text
设置 → 存储空间 → 应用程序 → 存储位置
```

这一层可以影响：

- App 本体放在哪个存储位置
- 部分 App 管理数据的位置

但它不保证：

- App 自己的下载数据
- 离线歌曲
- 缓存
- 自建文件夹

也会跟着搬。

一句话总结：

> App 安装位置、App 显示的下载路径、App 真正写入的离线数据路径，可能是三套不同逻辑。

## 主流程：本文最后确认的可用方法

本文最后确认的可用路线不是 root，也不是要求普通读者先格式化 SD 卡，而是：

1. 开启 USB 调试
2. 用 ADB 做观察与验证
3. 安装、执行并关闭目标 App
4. 通过 Sony UI 手动搬移 App
5. 再用 ADB 验证 App 本体与离线数据是否真的跟着搬移

### Step 1：确认系统是否看得到 SD 卡

```bash
adb shell sm list-disks
adb shell df -h
```

这一步的目的不是重建存储，而是确认：

- 系统有看到外接 SD 卡
- 内置与 SD 卡当前的容量结构
- 后续验证前的基准状态

注意：`sm list-disks` 显示的 `disk:179,192` 是 StorageManager 的磁盘设备代号，不是挂载路径，也不是内置存储本体。

### Step 2：先在随身听安装 QQ 音乐

本文案例中，QQ 音乐一开始是安装在随身听本体上，而不是先默认放到 SD 卡。 这一步很重要，因为后续要观察的是：

- 安装后 package 的实际位置
- 执行过一次后是否建立对应数据
- 手动搬移后 package 与数据是否同时跟着变化

### Step 3：执行一次 App，然后关闭

在安装完成后，先打开 QQ 音乐，让它完成必要初始化，再将 App 关闭。

这个顺序不能省略，因为没有先执行一次 App，就不会有可供系统管理的对应 package 状态与可观察的数据行为。

### Step 4：从 Sony UI 手动搬移 App

进入：

```text
设置 → 存储空间 → 应用程序 → 存储位置
```

若画面允许选择存储位置，手动把 QQ 音乐搬移到 SD 卡方向的存储位置。

这一步是本研究的核心操作。因为本文最后确认的可用方案不是高风险重建存储，而是确认 Sony Walkman 在现有系统下，是否已经提供可操作的 App 搬移入口。

### Step 5：用 ADB 验证 App 本体是否真的搬过去

先查 volume UUID：

```bash
adb shell dumpsys package com.tencent.qqmusic | findstr volumeUuid
```

再查 APK 路径：

```bash
adb shell pm path com.tencent.qqmusic
```

若结果显示：

- `volumeUuid` 指向外接卷
- `pm path` 指向对应的外接存储路径

代表 App 本体搬移成功。

### Step 6：验证离线数据是否跟着搬移

下载前先记录：

```bash
adb shell df -h /storage/emulated
adb shell df -h /mnt/expand/<UUID>
```

下载固定量离线歌曲后，再跑一次相同命令。

判读规则如下：

- 若 `/storage/emulated` 增加、`/mnt/expand/<UUID>` 几乎不变  
  → 代表离线数据仍主要写在内置共享存储层

- 若 `/mnt/expand/<UUID>` 增加  
  → 代表离线数据确实进入外接卷

## QQ 音乐案例：实测结果

本文案例的实际顺序是：

1. 安装 QQ 音乐
2. 打开并执行 App
3. 关闭 App
4. 到 Sony UI 手动搬移 App 到 SD 卡
5. 再用 ADB 验证 package 状态与数据写入位置

### App 本体验证

若 `dumpsys package` 显示的 `volumeUuid` 指向外接卷，且 `pm path` 指向外接路径，代表 QQ 音乐 App 本体已搬移成功。

### 离线数据验证

下载前后对照 `df -h`：

- 若 `/storage/emulated` 的空间减少
- 而 `/mnt/expand/<UUID>` 几乎没有变化

则表示：

> QQ 音乐的 App 本体搬移成功，但离线歌曲数据没有跟着搬移。

这就是本研究最重要的结论。

## 最终可用解法

虽然系统层没有真正把 QQ 音乐的离线下载数据自动导到 1TB，但最后仍找到一个实用上的可行方案。

### 实务作法
- 先让 QQ 音乐下载
- 再把音乐数据文件夹搬到 SD 卡上的音乐区，例如：

```text
SD卡/Music/qqmusic/
```

### 本质
Android 11 这里的限制比较像是：

- 限制你“写哪里”
- 但在很多情况下，不一定完全限制你“从哪里读”

所以最后形成的可用模式是：

```text
下载 -> 128GB
存放 -> 1TB
播放 -> 从 1TB 读
```

这不是系统层完全整合成功，但对音乐机用途来说，已经是可用且稳定的方案。

---

## 延伸：把 Walkman 当成纯音乐机

既然最终定位是音乐机，而不是一般 Android 手机，那后续最合理的存储策略应该是：

### 128GB 内置
保留给：

- Android 系统
- App 本体
- 必要缓存
- 少量临时数据

### 1TB SD 卡
保留给：

- 本地音乐主库
- 搬移后的音乐数据
- 长期收藏曲库

也就是：

> 128GB 当系统盘，1TB 当音乐盘。

这句话成立的前提是：

- 你接受下载层与存放层分离
- 不再要求 QQ 音乐天生自动把所有离线歌曲直接写进 1TB

## 最终结论

本次研究可收敛成五点：

1. Sony Walkman（Android 11）在 non-root 下可通过 UI 手动搬移 App
2. ADB 可用来验证 App 本体实际存储位置与数据写入位置
3. QQ 音乐 App 本体可搬，但离线歌曲数据不保证跟着搬移
4. 最终可行方案不是让系统完全合并存储，而是接受“系统层分离、使用层解决”
5. 这篇文章目前是已验证案例与可重现方法的整理，不是对所有 Android 11+ DAP 与所有串流 App 的最终定论

一句话总结：

> Sony Walkman 的 1TB SD 卡可以成为实际音乐盘，但不会自然成为所有 App 离线数据的主存储层；至少在 QQ 音乐案例中，App 本体搬移成功，离线歌曲搬移失败，最后必须通过数据搬移与使用策略调整来实际利用 1TB。

## 方法有没有错？审计判断

你的方法不是幻觉，也不能直接判定为错误。比较精确的说法是：

### 正确的部分
- 你有先安装 App，再执行、关闭、搬移，这是合理顺序
- 你有用 ADB 去验证 package 与存储行为，而不是只看 UI
- 你有把 UI 搬移与底层验证分开，这是正确做法

### 容易写偏的部分
- 若把 `sm set-force-adoptable true`、`sm partition ... private` 写成所有人必做主流程，就会偏掉
- 若把 `disk:179,192` 写成内置空间本体，就会偏掉
- 若把 UI 可搬移直接等同于离线数据一定会搬，就会偏掉

因此，本文应保留你的探索过程，但主流程必须改写成“UI 手动搬移 + ADB 验证”。

## Windows / MTP 为什么还是只看到 103GB

本案例中：

- Sony 的设置页可能会显示总容量约 1.13TB
- 但 Windows 通过 MTP 仍只看到约 103/104GB

原因是：

- MTP 主要 expose 的仍是共享存储层
- 外接扩展卷不一定会像一般 portable SD 一样直接当普通磁盘暴露给电脑

所以：

> Windows 看不到 1TB，不代表 1TB 没生效。它只是代表该外接卷不是以一般外接盘逻辑提供给 MTP。
