# Android 11+ DAP SD 卡儲存行為研究：以 Sony Walkman 為例

[返回主 README](../README.md) · [简体中文](README.zh-CN.md) · [English](README.en.md) · [FAQ](faq.md) · [相容性矩陣](compatibility.md)

## 前言

這篇文章不是單純記錄「QQ 音樂怎麼搬到 SD 卡」，而是要回答一個更底層的問題：

> 在 Android 11 之後，外接 SD 卡究竟能不能被 DAP（數位音樂播放器）上的 App 真正當作主要可用儲存空間？

Sony Walkman、iBasso、HiBy、FiiO 等 Android DAP，本質上都可能碰到同一類問題：

- 系統看得到 SD 卡，不等於 App 會把資料寫進去
- App 本體搬得動，不等於離線歌曲也會跟著搬
- UI 顯示容量變大，不等於主共享儲存空間真的被整合

因此，本文雖然以 **Sony Walkman + Android 11 + QQ 音樂** 為實測案例，但真正要處理的是 **Android 11+ DAP 的儲存分層問題**。

---

## 問題定義

這篇文章主要要解決的是以下幾類實際問題：

1. **SD 卡明明有被系統辨識，為什麼 App 還是在吃內建儲存？**
2. **App 明明可以搬移，為什麼離線歌曲或快取沒有跟著搬？**
3. **為什麼 Windows / MTP 還是只看到約 103GB，而不是完整 1TB？**
4. **哪些步驟只是研究探索，哪些才是一般使用者可重現的方法？**
5. **在 non-root 前提下，到底有沒有一條穩定、可驗證、可長期使用的路線？**

本文的目標不是把所有現象混成一句話，而是把「系統層」「UI 層」「App 資料層」拆開。

## 核心結論先行

先講結論，避免讀者在前面步驟中混淆：

1. **Sony Walkman 的外接 1TB SD 卡可以被系統辨識，也可以成為 App 可選擇的儲存位置。**
2. **QQ 音樂 App 本體可以透過系統 UI 手動搬移。**
3. **但 QQ 音樂的離線歌曲資料不一定跟著搬移。**
4. **本文最後確認的可用路線是：USB 偵錯 + ADB 驗證 + Sony UI 手動搬移 App。**
5. **本文不把 root 當成前提，也不把重新格式化 SD 卡當成一般讀者必做主流程。**

一句話總結：

> **App 本體搬移成功，不代表 App 下載資料搬移成功。**

## 研究目的

本次研究的目的有三個：

### 1. 解決容量限制
Sony Walkman 內建儲存約 128GB，對高音質音樂、串流離線資料與長期使用來說偏小。希望讓 1TB SD 卡成為主要音樂承載空間。

### 2. 解決寫入壓力集中在內建儲存的問題
如果大量下載、刪除、重建快取都長期寫在內建儲存，內建空間會同時承受容量與寫入壓力。因此希望把主要音樂資料流量導向可替換的 1TB SD 卡。

### 3. 驗證 Android 11 儲存分層在 DAP 上的實際影響
直覺上，1TB SD 卡插進去之後，應該可以讓音樂 App 直接使用大容量儲存。但實際上，Android 11 的儲存分層與 OEM 設計，可能讓「可見容量」與「可實際使用容量」不同。

## 底層模型：先分清楚三層

在正式操作前，必須先把 Android 11 的儲存結構拆開看。

### 1. 共享儲存層
典型路徑：

```text
/storage/emulated/0
```

這一層通常是：

- 使用者在檔案管理器中熟悉的「內部共用儲存空間」
- App 最常顯示給使用者看的下載路徑
- Windows 透過 MTP 最容易看到的那一層

很多人會誤以為這一層就是全部儲存，但它其實只是 App 和使用者最常接觸到的一層。

### 2. 擴充卷 / 外接卷層
典型路徑：

```text
/mnt/expand/<UUID>
```

這一層通常是：

- 外接儲存被系統掛載後的擴充卷路徑
- Package Manager 可能把 App 本體導向這裡
- 一般使用者不一定能直接透過檔案總管操作

這一層是系統底層真正看到的外接卷。

### 3. App Storage 管理層
這一層不是單一路徑，而是系統 UI 提供的管理入口，例如：

```text
設定 → 儲存空間 → 應用程式 → 儲存位置
```

這一層可以影響：

- App 本體放在哪個儲存位置
- 部分 App 管理資料的位置

但它不保證：

- App 自己的下載資料
- 離線歌曲
- 快取
- 自建資料夾

也會跟著搬。

一句話總結：

> App 安裝位置、App 顯示的下載路徑、App 真正寫入的離線資料路徑，可能是三套不同邏輯。

## 主流程：本文最後確認的可用方法

本文最後確認的可用路線不是 root，也不是要求一般讀者先格式化 SD 卡，而是：

1. 開啟 USB 偵錯
2. 用 ADB 做觀察與驗證
3. 安裝、執行並關閉目標 App
4. 透過 Sony UI 手動搬移 App
5. 再用 ADB 驗證 App 本體與離線資料是否真的跟著搬移

### Step 1：確認系統是否看得到 SD 卡

```bash
adb shell sm list-disks
adb shell df -h
```

這一步的目的不是重建儲存，而是確認：

- 系統有看到外接 SD 卡
- 內建與 SD 卡目前的容量結構
- 後續驗證前的基準狀態

注意：`sm list-disks` 顯示的 `disk:179,192` 是 StorageManager 的磁碟裝置代號，不是掛載路徑，也不是內建儲存本體。

### Step 2：先在隨身聽安裝 QQ 音樂

本文案例中，QQ 音樂一開始是安裝在隨身聽本體上，而不是先預設放到 SD 卡。這一步很重要，因為後續要觀察的是：

- 安裝後 package 的實際位置
- 執行過一次後是否建立對應資料
- 手動搬移後 package 與資料是否同時跟著變化

### Step 3：執行一次 App，然後關閉

在安裝完成後，先開啟 QQ 音樂，讓它完成必要初始化，再將 App 關閉。

這個順序不能省略，因為沒有先執行一次 App，就不會有可供系統管理的對應 package 狀態與可觀察的資料行為。

### Step 4：從 Sony UI 手動搬移 App

進入：

```text
設定 → 儲存空間 → 應用程式 → 儲存位置
```

若畫面允許選擇儲存位置，手動把 QQ 音樂搬移到 SD 卡方向的儲存位置。

這一步是本研究的核心操作。因為本文最後確認的可用方案不是高風險重建儲存，而是確認 Sony Walkman 在現有系統下，是否已經提供可操作的 App 搬移入口。

### Step 5：用 ADB 驗證 App 本體是否真的搬過去

先查 volume UUID：

```bash
adb shell dumpsys package com.tencent.qqmusic | findstr volumeUuid
```

再查 APK 路徑：

```bash
adb shell pm path com.tencent.qqmusic
```

若結果顯示：

- `volumeUuid` 指向外接卷
- `pm path` 指向對應的外接儲存路徑

代表 App 本體搬移成功。

### Step 6：驗證離線資料是否跟著搬移

下載前先記錄：

```bash
adb shell df -h /storage/emulated
adb shell df -h /mnt/expand/<UUID>
```

下載固定量離線歌曲後，再跑一次相同指令。

判讀規則如下：

- 若 `/storage/emulated` 增加、`/mnt/expand/<UUID>` 幾乎不變  
  → 代表離線資料仍主要寫在內建共享儲存層

- 若 `/mnt/expand/<UUID>` 增加  
  → 代表離線資料確實進入外接卷

## QQ 音樂案例：實測結果

本文案例的實際順序是：

1. 安裝 QQ 音樂
2. 開啟並執行 App
3. 關閉 App
4. 到 Sony UI 手動搬移 App 到 SD 卡
5. 再用 ADB 驗證 package 狀態與資料寫入位置

### App 本體驗證

若 `dumpsys package` 顯示的 `volumeUuid` 指向外接卷，且 `pm path` 指向外接路徑，代表 QQ 音樂 App 本體已搬移成功。

### 離線資料驗證

下載前後對照 `df -h`：

- 若 `/storage/emulated` 的空間減少
- 而 `/mnt/expand/<UUID>` 幾乎沒有變化

則表示：

> QQ 音樂的 App 本體搬移成功，但離線歌曲資料沒有跟著搬移。

這就是本研究最重要的結論。

## 最終可用解法

雖然系統層沒有真正把 QQ 音樂的離線下載資料自動導到 1TB，但最後仍找到一個實用上的可行方案。

### 實務作法
- 先讓 QQ 音樂下載
- 再把音樂資料夾搬到 SD 卡上的音樂區，例如：

```text
SD卡/Music/qqmusic/
```

### 本質
Android 11 這裡的限制比較像是：

- 限制你「寫哪裡」
- 但在很多情況下，不一定完全限制你「從哪裡讀」

所以最後形成的可用模式是：

```text
下載 -> 128GB
存放 -> 1TB
播放 -> 從 1TB 讀
```

這不是系統層完全整合成功，但對音樂機用途來說，已經是可用且穩定的方案。

---

## 延伸：把 Walkman 當成純音樂機

既然最終定位是音樂機，而不是一般 Android 手機，那後續最合理的儲存策略應該是：

### 128GB 內建
保留給：

- Android 系統
- App 本體
- 必要快取
- 少量臨時資料

### 1TB SD 卡
保留給：

- 本地音樂主庫
- 搬移後的音樂資料
- 長期收藏曲庫

也就是：

> 128GB 當系統盤，1TB 當音樂盤。

這句話成立的前提是：

- 你接受下載層與存放層分離
- 不再要求 QQ 音樂天生自動把所有離線歌曲直接寫進 1TB

## 最終結論

本次研究可收斂成五點：

1. Sony Walkman（Android 11）在 non-root 下可透過 UI 手動搬移 App
2. ADB 可用來驗證 App 本體實際儲存位置與資料寫入位置
3. QQ 音樂 App 本體可搬，但離線歌曲資料不保證跟著搬移
4. 最終可行方案不是讓系統完全合併儲存，而是接受「系統層分離、使用層解決」
5. 這篇文章目前是已驗證案例與可重現方法的整理，不是對所有 Android 11+ DAP 與所有串流 App 的最終定論

一句話總結：

> Sony Walkman 的 1TB SD 卡可以成為實際音樂盤，但不會自然成為所有 App 離線資料的主儲存層；至少在 QQ 音樂案例中，App 本體搬移成功，離線歌曲搬移失敗，最後必須透過資料搬移與使用策略調整來實際利用 1TB。

## 方法有沒有錯？審計判斷

你的方法不是幻覺，也不能直接判定為錯誤。比較精確的說法是：

### 正確的部分
- 你有先安裝 App，再執行、關閉、搬移，這是合理順序
- 你有用 ADB 去驗證 package 與儲存行為，而不是只看 UI
- 你有把 UI 搬移與底層驗證分開，這是正確做法

### 容易寫偏的部分
- 若把 `sm set-force-adoptable true`、`sm partition ... private` 寫成所有人必做主流程，就會偏掉
- 若把 `disk:179,192` 寫成內建空間本體，就會偏掉
- 若把 UI 可搬移直接等同於離線資料一定會搬，就會偏掉

因此，本文應保留你的探索過程，但主流程必須改寫成「UI 手動搬移 + ADB 驗證」。

## Windows / MTP 為什麼還是只看到 103GB

本案例中：

- Sony 的設定頁可能會顯示總容量約 1.13TB
- 但 Windows 透過 MTP 仍只看到約 103/104GB

原因是：

- MTP 主要 expose 的仍是共享儲存層
- 外接擴充卷不一定會像一般 portable SD 一樣直接當普通磁碟暴露給電腦

所以：

> Windows 看不到 1TB，不代表 1TB 沒生效。它只是代表該外接卷不是以一般外接碟邏輯提供給 MTP。
