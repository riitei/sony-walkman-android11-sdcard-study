# 常見問題（FAQ）

這份 FAQ 用來回答 **Android 11+ DAP SD 卡儲存行為** 最常見、也最容易被搜尋到的問題。

[返回主 README](../README.md) · [繁中正文](README.zh-TW.md) · [简体中文 FAQ](faq.zh-CN.md) · [English FAQ](faq.md) · [相容性矩陣](compatibility.md)

---

## 1. SD 卡明明有被辨識，為什麼 App 還是在吃內建儲存？

因為這不是同一層問題。

在 Android 11+ 上，至少有三件事可能分開運作：

- **系統**看得到 SD 卡
- **UI** 允許搬移 App
- **App 自己的離線資料** 仍然寫在內建共享儲存層

所以：

> **SD 卡被辨識，不等於離線歌曲會自動存進 SD 卡。**

---

## 2. App 已經搬移成功，為什麼資料沒有跟著搬？

因為 **App 本體位置** 和 **App 資料位置** 本來就不保證相同。

一個 App 的安裝包可以搬到外接卷，但離線歌曲、快取或下載資料，仍可能留在：

```text
/storage/emulated/0
```

這正是本 repo 想釐清的核心問題。

---

## 3. 把 App 搬到 SD 卡，離線音樂也會一起搬嗎？

**不一定。**

這正是本文的主要發現之一。

已驗證案例：Sony Walkman + QQ 音樂

- App 本體搬移：**有**
- 離線歌曲跟著搬移：**沒有**

所以正確說法是：

> **App 搬移成功，不代表離線資料搬移成功。**

---

## 4. 為什麼 Windows / MTP 還是只看到大約 103GB，不是完整 1TB？

因為 Windows 經由 MTP 看到的，通常是 **共享儲存層**，不是每一個底層擴充卷。

在本文案例裡：

- 裝置設定頁可能顯示總容量約 1.13TB
- 但 Windows 仍然只看到約 103/104GB

所以：

> **Windows 看不到 1TB，不代表 SD 卡沒生效；只是代表它不是用一般外接碟的方式暴露給 MTP。**

---

## 5. `/storage/emulated/0` 是什麼？

它是 Android 最常見的 **內部共用儲存層**。

很多 App 的下載位置、很多檔案管理器看到的「內部儲存空間」，本質上都是這一層。

但它不代表整台裝置所有可用儲存都在這裡。

---

## 6. `/mnt/expand/<UUID>` 是什麼？

它是系統底層看到的 **擴充卷 / 外接卷路徑**。

在某些情況下，Package Manager 會把 App 本體放到這裡。

但即使 App 本體在這裡，也不保證離線歌曲或快取資料也會一起進來。

---

## 7. `disk:179,192` 是什麼意思？

它是 Android `StorageManager` 回報的 **磁碟裝置代號**，常見於：

```bash
adb shell sm list-disks
```

它：

- **不是掛載路徑**
- **不是內建儲存本體**
- **只是系統用來識別磁碟的代號**

---

## 8. 一定要 root 嗎？

**不用。**

本 repo 驗證過的主流程是 **non-root**：

1. 開 USB 偵錯
2. 用 ADB 觀察與驗證
3. 安裝並初始化 App
4. 用系統 UI 手動搬移 App
5. 再用 ADB 驗證實際結果

---

## 9. 下面這些指令一定要跑嗎？

```bash
adb shell sm set-force-adoptable true
adb shell sm partition disk:179,192 private
adb reboot
adb shell sm list-volumes all
```

**不一定，而且不應該當成一般讀者主流程。**

這些命令屬於本文的 **研究歷程**，用途是：

- 觀察系統底層儲存行為
- 理解 adoptable / private volume
- 協助釐清 UI 和底層的差異

但它們：

- 可能涉及儲存重建
- 可能伴隨格式化風險
- 不是本文最後確認的穩定主流程

所以正確說法是：

> **這些命令有研究價值，但不是一般使用者必做操作。**

---

## 10. 這篇的主流程到底是 UI 還是指令？

對一般使用者來說，主流程是：

- **UI 手動搬移 App**
- 加上 **ADB 驗證**

也就是：

1. 安裝 App
2. 執行一次
3. 關閉
4. 進入：

```text
設定 → 儲存空間 → 應用程式 → 儲存位置
```

5. 手動搬移 App
6. 再用 ADB 看實際結果

一句話：

> **搬移動作是 UI 主導，驗證動作是 ADB 主導。**

---

## 11. 為什麼要先安裝 App、執行一次，再關閉？

因為如果沒有這個順序：

- App 可能還沒建立完整初始化狀態
- package 狀態可能還不完整
- 後面的 ADB 驗證就沒有可比對的基準

所以這不是多餘步驟，而是方法的一部分。

---

## 12. 為什麼 QQ 音樂可以搬，離線歌曲卻還是在內建？

因為 **App 本體** 和 **離線資料** 是不同層在管。

- 系統可以允許 App 本體搬移
- App 自己仍可能把離線歌曲寫進共享儲存層

這不是矛盾，而是權責分離。

---

## 13. 「修改系統設定」權限有關嗎？

QQ 音樂可能有這樣的入口：

```text
應用程式資訊 → 進階 → 修改系統設定
```

就目前本文的驗證結果來看：

**沒有足夠證據** 顯示這個權限會直接決定離線歌曲能不能搬到外接儲存。

所以目前只能把它列為觀察項，不能直接當主因。

---

## 14. 這個問題靠 AI 能解決嗎？

AI 能幫你的地方有：

- 把問題拆成不同層
- 設計可重現的驗證流程
- 解讀 ADB 結果
- 分辨 UI 假象與底層真實行為
- 幫你整理成研究文件與相容性紀錄

但 AI 不能保證：

- Android 底層會照你想要的方式運作
- 每個 App 都會把離線資料搬過去
- 每台 DAP 都一樣

所以最誠實的答案是：

> **AI 可以幫你理解、驗證、縮小問題，但不能保證替你改寫 Android 規則或 App 實作限制。**

---

## 15. 現在最實際可行的解法是什麼？

以 Sony Walkman + QQ 音樂案例來說，最實用的解法是：

- 把內建儲存當成 **系統 / App 盤**
- 把 1TB SD 卡當成 **音樂資料盤**
- 接受「下載層」與「存放層」可能分離
- 必要時手動搬移音樂資料夾

簡化寫法就是：

```text
下載 -> 內建儲存
存放 -> SD 卡
播放 -> 從 SD 卡讀
```

**不是。**
