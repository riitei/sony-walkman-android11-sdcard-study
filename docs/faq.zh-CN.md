# 常见问题（FAQ）

这份 FAQ 整理测试 **Android 11+ DAP SD 卡存储行为** 时最常遇到的问题。

[返回主 README](../README.md) · [繁體中文 FAQ](faq.zh-TW.md) · [English FAQ](faq.md) · [简中正文](README.zh-CN.md) · [相容性矩阵](compatibility.md)

---

## 1. SD 卡明明已经被识别，为什么 App 还是在占用内置存储？

因为 SD 卡识别和 App 数据写入不是同一层。

在 Android 11+ 上，系统可以看得到 SD 卡，设置画面也可以允许搬移 App，但 App 自己的离线歌曲或缓存仍可能写在内置共享存储层。这份 repo 会把这些层分开看。

---

## 2. App 已经搬移成功，为什么数据没有跟着搬？

App 本体位置和 App 离线数据位置不一定相同。安装包可以搬到外接卷，但下载数据或缓存仍可能留在：

```text
/storage/emulated/0
```

这就是 Sony Walkman + QQ 音乐案例中观察到的主要现象。

---

## 3. 把 App 搬到 SD 卡，离线音乐也会一起搬吗？

不一定。

已验证的 Walkman + QQ 音乐测试结果是：

```text
App 本体搬移 -> 有
离线歌曲跟着搬移 -> 没有
```

所以 App 搬移成功不能直接当成离线数据搬移成功。

---

## 4. 为什么 Windows / MTP 还是只看到大约 103GB，不是完整 1TB？

Windows / MTP 通常看到的是共享存储层，不是 Android 底下每一个外接或扩展卷。

在这次测试中，Walkman 设置页可能显示较大的总容量，但 Windows 仍主要看到内置共享存储那一侧。这不代表 SD 卡没生效，只代表它不是用一般外接盘的方式暴露给 MTP。

---

## 5. `/storage/emulated/0` 是什么？

它是用户最常在文件管理器看到的内部共享存储层。很多 App 也会把它显示成下载位置。

它很重要，但不代表整台设备所有可用存储都在这里。

---

## 6. `/mnt/expand/<UUID>` 是什么？

它是 Android 底层看到的外接卷或扩展卷路径。在某些情况下，Package Manager 可能把 App 本体放到这里。

但即使 App 本体在这里，也不代表离线歌曲或缓存数据一定会一起进来。

---

## 7. `disk:179,192` 是什么意思？

它是 Android `StorageManager` 回报的磁盘设备代号，常见于：

```bash
adb shell sm list-disks
```

它不是挂载路径，也不是内置存储本体，只是系统用来识别磁盘的代号。

---

## 8. 一定要 root 吗？

不用。這份 repo 验证过的主流程是 non-root。

一般路线是：开 USB 调试，用 ADB 观察，安装并执行一次 App，用系统 UI 手动搬移 App，最后再用 ADB 验证 App 本体和数据行为。

---

## 9. 下面这些指令一定要跑吗？

```bash
adb shell sm set-force-adoptable true
adb shell sm partition disk:179,192 private
adb reboot
adb shell sm list-volumes all
```

不用。它们属于研究历程，不是普通读者的主流程。

这些指令有助于观察底层存储行为，但可能牵涉存储重建或格式化风险。这份 repo 的稳定主线仍是 UI 手动搬移 App，再用 ADB 验证。

---

## 10. 这篇的主流程到底是 UI 还是指令？

搬移动作是 UI 主导，验证动作是 ADB 主导。

实际操作是先安装 App、执行一次、关闭，再进入：

```text
设置 → 存储空间 → 应用程序 → 存储位置
```

手动搬移后，再用 ADB 看实际变化。

---

## 11. 为什么要先安装 App、执行一次，再关闭？

这样 App 才有一个比较完整的初始化状态。没有先执行一次，package 状态和数据行为可能不完整，后面的 ADB 对比也比较没有意义。

---

## 12. 为什么 QQ 音乐可以搬，离线歌曲却还在内置？

因为系统能搬的是 App 本体，App 自己仍可能把离线歌曲写进共享存储层。

这不是矛盾，而是 App 本体位置和 App 数据行为分开。

---

## 13. “修改系统设置”权限有关系吗？

QQ 音乐可能有这样的入口：

```text
应用程序信息 → 高级 → 修改系统设置
```

就这次存储行为测试来看，目前没有足够证据显示这个权限会直接决定离线歌曲能不能搬到外接存储。它可以记录为观察项，但不适合直接当主因。

---

## 14. 这个问题靠 AI 能解决吗？

AI 可以帮忙拆层、设计验证流程、解读 ADB 结果，也可以把测试记录整理成比较清楚的文件。

但 AI 不能改写 Android 的存储规则，也不能保证每个 App 都会把离线数据写到用户想要的位置。这件事最后仍要靠设备端验证。

---

## 15. 现在最实际可行的解法是什么？

以 Sony Walkman + QQ 音乐案例来说，目前比较实际的角色分工是：

```text
下载 -> 内置存储
存放 -> SD 卡
播放 -> 从 SD 卡读
```

也就是把内置存储当系统 / App 盘，把 1TB SD 卡当音乐资料盘。必要时仍需要手动搬移音乐数据文件夹。

---

## 16. 这个 repo 是不是对所有 DAP 和所有串流 App 都成立？

不是。它目前是一个 Sony Walkman 已验证案例、一套可重现方法，以及后续相容性回报入口。

其他设备与 App 请看 [相容性矩阵](compatibility.md)。目前没验证过的项目，都应视为待验证问题，不是既定事实。
