# 常见问题（FAQ）

这份 FAQ 用来回答 **Android 11+ DAP SD 卡存储行为** 最常见、也最容易被搜索到的问题。

[返回主 README](../README.md) · [繁體中文 FAQ](faq.zh-TW.md) · [English FAQ](faq.md) · [简中正文](README.zh-CN.md) · [相容性矩阵](compatibility.md)

---

## 1. SD 卡明明已经被识别，为什么 App 还是在占用内置存储？

因为这不是同一层问题。

在 Android 11+ 上，至少有三件事可能分开运作：

- **系统**看得到 SD 卡
- **UI** 允许搬移 App
- **App 自己的离线数据** 仍然写在内置共享存储层

所以：

> **SD 卡被识别，不等于离线歌曲会自动存进 SD 卡。**

---

## 2. App 已经搬移成功，为什么数据没有跟着搬？

因为 **App 本体位置** 和 **App 数据位置** 本来就不保证相同。

一个 App 的安装包可以搬到外接卷，但离线歌曲、缓存或下载数据，仍可能留在：

```text
/storage/emulated/0
```

这正是本 repo 想厘清的核心问题。

---

## 3. 把 App 搬到 SD 卡，离线音乐也会一起搬吗？

**不一定。**

这正是本文的主要发现之一。

已验证案例：Sony Walkman + QQ 音乐

- App 本体搬移：**有**
- 离线歌曲跟着搬移：**没有**

所以正确说法是：

> **App 搬移成功，不代表离线数据搬移成功。**

---

## 4. 为什么 Windows / MTP 还是只看到大约 103GB，不是完整 1TB？

因为 Windows 经由 MTP 看到的，通常是 **共享存储层**，不是每一个底层扩展卷。

在本文案例里：

- 设备设置页可能显示总容量约 1.13TB
- 但 Windows 仍然只看到约 103/104GB

所以：

> **Windows 看不到 1TB，不代表 SD 卡没生效；只是代表它不是用一般外接盘的方式暴露给 MTP。**

---

## 5. `/storage/emulated/0` 是什么？

它是 Android 最常见的 **内部共享存储层**。

很多 App 的下载位置、很多文件管理器看到的“内部存储空间”，本质上都是这一层。

但它不代表整台设备所有可用存储都在这里。

---

## 6. `/mnt/expand/<UUID>` 是什么？

它是系统底层看到的 **扩展卷 / 外接卷路径**。

在某些情况下，Package Manager 会把 App 本体放到这里。

但即使 App 本体在这里，也不保证离线歌曲或缓存数据也会一起进来。

---

## 7. `disk:179,192` 是什么意思？

它是 Android `StorageManager` 回报的 **磁盘设备代号**，常见于：

```bash
adb shell sm list-disks
```

它：

- **不是挂载路径**
- **不是内置存储本体**
- **只是系统用来识别磁盘的代号**

---

## 8. 一定要 root 吗？

**不用。**

本 repo 验证过的主流程是 **non-root**：

1. 开 USB 调试
2. 用 ADB 观察与验证
3. 安装并初始化 App
4. 用系统 UI 手动搬移 App
5. 再用 ADB 验证实际结果

---

## 9. 下面这些指令一定要跑吗？

```bash
adb shell sm set-force-adoptable true
adb shell sm partition disk:179,192 private
adb reboot
adb shell sm list-volumes all
```

**不一定，而且不应该当成普通读者主流程。**

这些命令属于本文的 **研究历程**，用途是：

- 观察系统底层存储行为
- 理解 adoptable / private volume
- 协助厘清 UI 和底层的差异

但它们：

- 可能涉及存储重建
- 可能伴随格式化风险
- 不是本文最后确认的稳定主流程

所以正确说法是：

> **这些命令有研究价值，但不是普通用户必做操作。**

---

## 10. 这篇的主流程到底是 UI 还是指令？

对普通用户来说，主流程是：

- **UI 手动搬移 App**
- 加上 **ADB 验证**

也就是：

1. 安装 App
2. 执行一次
3. 关闭
4. 进入：

```text
设置 → 存储空间 → 应用程序 → 存储位置
```

5. 手动搬移 App
6. 再用 ADB 看实际结果

一句话：

> **搬移动作是 UI 主导，验证动作是 ADB 主导。**

---

## 11. 为什么要先安装 App、执行一次，再关闭？

因为如果没有这个顺序：

- App 可能还没建立完整初始化状态
- package 状态可能还不完整
- 后面的 ADB 验证就没有可比对的基准

所以这不是多余步骤，而是方法的一部分。

---

## 12. 为什么 QQ 音乐可以搬，离线歌曲却还在内置？

因为 **App 本体** 和 **离线数据** 是不同层在管理。

- 系统可以允许 App 本体搬移
- App 自己仍可能把离线歌曲写进共享存储层

这不是矛盾，而是权责分离。

---

## 13. “修改系统设置”权限有关系吗？

QQ 音乐可能有这样的入口：

```text
应用程序信息 → 高级 → 修改系统设置
```

就目前本文的验证结果来看：

**没有足够证据** 显示这个权限会直接决定离线歌曲能不能搬到外接存储。

所以目前只能把它列为观察项，不能直接当成主因。

---

## 14. 这个问题靠 AI 能解决吗？

AI 能帮你的地方有：

- 把问题拆成不同层
- 设计可重现的验证流程
- 解读 ADB 结果
- 分辨 UI 假象与底层真实行为
- 帮你整理成研究文件与相容性记录

但 AI 不能保证：

- Android 底层会照你想要的方式运作
- 每个 App 都会把离线数据搬过去
- 每台 DAP 都一样

所以最诚实的答案是：

> **AI 可以帮你理解、验证、缩小问题，但不能保证替你改写 Android 规则或 App 实作限制。**

---

## 15. 现在最实际可行的解法是什么？

以 Sony Walkman + QQ 音乐案例来说，最实用的解法是：

- 把内置存储当成 **系统 / App 盘**
- 把 1TB SD 卡当成 **音乐资料盘**
- 接受“下载层”与“存放层”可能分离
- 必要时手动搬移音乐数据文件夹

简化写法就是：

```text
下载 -> 内置存储
存放 -> SD 卡
播放 -> 从 SD 卡读
```

## 16. 这个 repo 是不是对所有 DAP 和所有串流 App 都成立？

**不是。**

它目前是：

- 一个 **Sony Walkman 已验证案例**
- 一套 **可重现方法**
- 一个 **后续可扩充的研究入口**

其他设备与其他 App，请看：

- [相容性矩阵](compatibility.md)

目前没验证过的项目，都应视为**待验证问题**，不是既定事实。
